TCP(传输控制协议)总结

TCP的概念和特点

  • TCP是**运输层的面向连接的可靠的**运输协议。

  • TCP连接由两台主机上的缓存、变量和与进程相连的套接字组成，是一条逻辑上的连接。两台主机之间的网络节点没有为该连接分配任何缓存或变量。

  • TCP连接提供全双工服务，UDP可以是全双工的。

  • TCP在网络层IP不可靠的尽力而为服务之上，依靠确认、序列号、RTT估算、超时和重传等机制，创建了一种可靠的数据传输服务。UDP没有这些机制，不提供可靠性。

  • TCP依靠滑动窗口实现了流量控制服务，以消除发送方使得接收方缓存溢出的可能性，UDP不提供流量控制。流量控制是一个速度匹配服务，即发送方的发送速率与接收方应用程序的读取速率相匹配。

    滑动窗口的大小意味着接收方还有多大的缓冲区可以用于 接收数据。发送方可以通过滑动窗口的大小来确定应该发送多少字节的数据。当滑动窗口为 0 时，发送方一般不能再发送数据报。 滑动窗口是 TCP 中实现诸如 ACK 确认、流量控制、拥塞控制的承载结构。

三次握手

三次握手其实就是指在建立TCP连接时需要客户端和服务器端总共发送三个数据包，主要作用是确认服务器端和客户端的发送和接受能力是否正常，指定自己的初始序列号为后面的可靠性传输作准备。实质上其实就是连接服务器指定端口，建立TCP连接，并同步连接双方的序列号和确认号，交换TCP窗口大小信息

Client: “喂，你能听到我说话吗？” （第一次握手）

Server: “能听到，你能听到吗？” （第二次握手）

Client: “我也能听到” （第三次握手）

Q: 三次握手的流程

首先服务器通过调用socket,bind和listen这三个函数，切换到LISTEN状态，准备好接受外来的连接，这一过程称为被动打开

  1. ==第一次握手：==客户端通过调用connect发起主动打开，发送一个SYN(告诉服务器客户端消息的初始序列号seq=x)，并切换到SYN-SENT状态
  2. ==第二次握手：==服务器必须ACK客户端的SYN，自己也得发一个SYN(告诉客户端服务器消息的初始序列号seq=y)，并切换到SYN-RCVD状态
  3. ==第三次握手：==客户端必须ACK服务器的SYN，并切换到ESTABLISHED状态
  4. 服务器收到客户端的ACK后，切换到ESTABLISHED状态

注意：ACK中的确认号是发送这个ACK的一端所期待的下一个序列号，所以第二步和第三步里ack=x+1和ack=y+1。第三次握手可以携带数据，前两次不可以。

Q: 为什么一定需要三次握手呢？两次可以吗？

第一次握手服务端可以看出 ：客户端的发送能力，和自己的接收能力处于正常状态

第二次握手客户端可以看出：服务端的发送和接收能力都正常，客户端的发送和接收能力也都正常，但服务端不知道客户端的接收能力是否正常。

第三次握手可以看出： 客户端的发送接收，服务端的发送接收能力均正常。

因为TCP连接是全双工的，至少要三次才能确认双方的接收与发送能力是否正常，才能保证双向连接都成功。

TCP是基于IP协议的，而IP协议是有路由的，IP协议不能够保证先发送的数据先到达，这当中依赖于IP协议底层的网络质量，以及Client与Server之间的路由跳数。

如果是两次握手，会出现如下情况：

假设客户端发起连接请求，发送第一个SYN报文，称为SYN1。因为网络原因，SYN1滞留了。客户端超时重发连接请求，称为SYN2，因为网络原因，SYN2先到达服务器，服务器和客户端基于SYN2建立了连接。

然后SYN1到达了，此时服务端误认为客户端又发出一次新的连接请求，于是就向客户端发出ACK，同意建立连接，不采用三次握手，只要服务端发出确认，就建立新的连接了，此时客户端忽略服务端发来的确认，也不发送数据，则服务端一直等待客户端发送数据，浪费资源。

Q: 什么是半连接队列？

服务器第一次收到客户端的 SYN 之后，就会处于 SYN_RCVD 状态，此时双方还没有完全建立其连接，服务器会把此种状态下请求连接放在一个队列里，我们把这种队列称之为半连接队列。

当然还有一个全连接队列，就是已经完成三次握手，建立起连接的就会放在全连接队列中。如果队列满了就有可能会出现丢包现象。

Q: SYN攻击是什么？

服务器端的资源分配是在二次握手时分配的，而客户端的资源是在完成三次握手时分配的**，所以服务器容易受到SYN洪泛攻击。SYN攻击就是Client在短时间内伪造大量不存在的IP地址，并向Server不断地发送SYN包，Server则回复确认包，并等待Client确认，由于源地址不存在，因此Server需要不断重发直至超时，这些伪造的SYN包将长时间占用未连接队列，导致正常的SYN请求因为队列满而被丢弃，从而引起网络拥塞甚至系统瘫痪。SYN 攻击是一种典型的 DoS/DDoS 攻击。

检测 SYN 攻击非常的方便，当你在服务器上看到大量的半连接状态时，特别是源IP地址是随机的，基本上可以断定这是一次SYN攻击。在 Linux/Unix 上可以使用系统自带的 netstats 命令来检测 SYN 攻击。

常见的防御 SYN 攻击的方法有如下几种：

  • 缩短超时（SYN Timeout）时间
  • 增加最大半连接数
  • 过滤网关防护
  • SYN cookies技术

四次挥手

四次挥手，顾名思义就是客户端和服务端四个步骤的释放连接，断开连接需要发送四个包，别名连接终止协议。由于TCP连接是全双工的，因此每个方向都必须单独进行关闭。这原则是当一方完成它的数据发送任务后就能发送一个FIN来终止这个方向的连接。收到一个 FIN只意味着这一方向上没有数据流动，一个TCP连接在收到一个FIN后仍能发送数据。首先进行关闭的一方将执行主动关闭，而另一方执行被动关闭。

假如你在外边玩，此时你拨通了妈妈的电话
1.你：“妈我饿了，能给我做饭吃吗”
2.你的妈妈：“好，我现在准备准备，等会做好了喊你”
——————————————————————–做饭的过程————————————————————————
3.你的妈妈：“儿，饭做好了，该下面条了，什么时候回来呀”
4.你：“好嘞妈，你先下面条吧，我正在上楼”

第二个步骤和第三个步骤不能合并，因为第二个步骤妈妈确定要做饭了，先回复你一声让你知道妈妈已收到。但还没有做好饭，所有等饭做好了，再通知你一切都准备好了。如果合并第二步和第三步，因为做饭要很长时间，这段时间你都没有收到妈妈的回复，你不知道她有没有开始做饭，就会不停地问而收不到回复，然后你就去饭馆了，最后妈妈的饭也白做了。

这正是三次握手而四次挥手的原因，三次握手相当于把第二步和第三步合并了，因为不携带数据，ACK和SYN可以一块发送，四次挥手的ACK和FIN却得分开发送。

Q: 四次挥手的流程

两端都可以主动关闭，这里以客户端主动发起关闭为例

  1. ==第一次挥手：==客户端某个进程首先调用close，执行主动关闭，发送一个FIN，表示客户端数据发送完毕，并切换到FIN-WAIT-1状态
  2. ==第二次挥手：==服务器接收到FIN后进行ACK，执行被动关闭。把一个文件结束符传递给接收端应用进程，通知应用进程在相应连接上再无额外数据可以接收，并切换到CLOSE-WAIT状态。这时TCP处于半关闭状态，服务器仍可能发送数据到客户端，当客户端收到回复后，进入FIN-WAIT-2状态
  3. ==第三次挥手：==服务器的应用进程接收到文件结束符后调用close关闭其socket，这导致它的TCP也发送一个FIN，表示服务器端数据也发送完毕，并切换到LAST-ACK状态
  4. ==第四次挥手：==客户端接收这个最终FIN后对其进行ACK，并切换到TIME-WAIT状态。TIME-WAIT状态是为了等待足够的时间以确保远程TCP接收到连接中断请求的确认。

四次挥手中的FIN和三次握手中的SYN类似，其中都会包含一个序列号。而每个FIN的ACK确认号就是这个FIN的序列号加1

注意：这个时候由服务端到客户端的 TCP 连接并未释放掉，客户端需要经过时间等待计时器设置的时间 2MSL（一个报文的来回时间） 后才会进入CLOSED状态（这样做的目的是确保服务端收到自己的 ACK 报文。如果服务端在规定时间内没有收到客户端发来的 ACK 报文的话，服务端会重新发送 FIN 报文给客户端，客户端再次收到 FIN 报文之后，就知道之前的 ACK 报文丢失了，然后再次发送 ACK报文给服务端）。服务端收到 ACK 报文之后，就关闭连接了，处于 CLOSED 状态。

Q: 需要进入TIME-WAIT状态并等待2MSL时间的原因？

  1. TCP实现必须可靠地终止连接的两个方向(全双工关闭)，为了防⽌客户端最后⼀次发给服务器的ACK在⽹络中丢失以⾄于客户端关闭，⽽服务端并未关闭，导致资源的浪费，客户端必须进入TIME_WAIT状态。
  2. 等待最⼤的2MSL可以让本次连接的所有的⽹络包在链路上消失，以防造成不必要的⼲扰。
    　　如果客户端直接closed，然后⼜向服务端发起了⼀个新连接，我们不能保证这个新连接和刚关闭的连接的端⼝号是不同的。假设新连接和已经关闭的⽼端⼝号是⼀样的，如果前⼀次滞留的某些数据仍然在⽹络中，这些延迟数据会在新连接建⽴后到达服务端，所以socket就认为那个延迟的数据是属于新连接的，数据包就会发⽣混淆。所以客户端要在TIME_WAIT状态等待2倍的MSL，这样保证本次连接的所有数据都从⽹络中消失

Q: 为什么要四次挥手？三次不行吗？

因为握手的时候并没有数据传输，所以服务端的 SYN 和 ACK 报文可以一起发送，但是挥手的时候有数据在传输，所以ACK 和 FIN 报文不能同时发送，需要分两步，所以会比握手多一步。

因为服务端在接收到FIN, 往往不会立即返回FIN ，必须等到服务端所有的报文都发送完毕了，才能发FIN。因此先发一个ACK表示已经收到客户端的FIN，延迟一段时间才发FIN。这就造成了四次挥手。

如果是三次挥手会造成：
如果将服务端的两次挥手合为一次，等于说服务端将ACK和FIN的发送合并为一次挥手，这个时候长时间的延迟可能会导致客户端误以为FIN没有到达客户端，从而让客户端不断的重发FIN。所有只能第二次挥手先发送ACK确认接收到了客户端的数据，等服务器发送完了数据，再发送FIN包进行第三次挥手。

TCP状态转换

TCP拥塞控制

  1. 拥塞控制: 通过拥塞窗口来防止过多的数据注入网络，以避免网络中的路由器或者链路过载
  • 拥塞窗口cwnd是发送方维护的一个状态变量，根据网络拥塞程度变化
  • 发送窗口swnd = min(cwnd,rwnd)，也就是拥塞窗口和接收窗口中的最小值
  • 网络中没有出现拥塞时cwnd增大，出现拥塞时cwnd减小

拥塞控制与流量控制的区别？
拥塞控制是防止过多的数据注入到网络中，可以使网络中的路由器或链路不致过载，是一个全局性的过程。
流量控制是点对点通信量的控制，是一个端到端的问题，主要就是抑制发送端发送数据的速率，以便接收端来得及接收。

  1. 出现拥塞的标志: 重传计时器超时 或 接收到三个重复ACK
  2. TCP拥塞控制算法: 慢启动、拥塞避免、拥塞发生、快速恢复
    1. 慢启动：发送方每收到一个ACK，cwnd的大小增加一个MSS(指数增长)
      • 慢启动指不是增长慢，而是cwnd的初始值比较小(1个MSS)
      • 慢启动结束的三种方式：
        1. 发生超时丢包事件（即拥塞）。此时转为拥塞发生模式，执行超时重传。
        2. cwnd等于ssthresh时。此时转为拥塞避免模式。
        3. 检测到三个冗余ACK（即拥塞）。此时转为拥塞发生模式，执行快速重传。
    2. 拥塞避免：发送方每收到一个ACK，cwnd以线性方式增长
      • 此时cwnd增长速度会变慢，直到发生拥塞。
      • 与慢启动一样，发生拥塞时切换到拥塞发生模式。超时，则执行超时重传。三个冗余ACK，则执行快速重传。
    3. 拥塞发生：发生了拥塞，进行数据重传
      • 超时重传：由超时重传事件触发丢包时，执行该算法，有如下3个操作：
        1. ssthresh 设置为 cwnd/2
        2. cwnd重置为1
        3. 切换到慢启动模式
      • 快速重传：由三个冗余ACK触发丢包时，执行该算法，有如下3个操作：
        1. cwnd 设置为 cwnd/2
        2. ssthresh 设置为 cwnd
        3. 切换到快速恢复模式
    4. 快速恢复：一般和快速重传同时使用。TCP认为三个冗余ACK丢包事件没有超时丢包事件严重。步骤如下：
      1. cwnd = ssthresh + 3
      2. 重传丢失的数据包
      3. 若再收到冗余ACK，cwnd加1
      4. 若收到新数据的ACK，把cwnd设置为第一步中的ssthresh的值。因为此时说明发生冗余ACK时的丢包数据已经全部收到，恢复过程已结束，可以再次进入拥塞避免状态。
